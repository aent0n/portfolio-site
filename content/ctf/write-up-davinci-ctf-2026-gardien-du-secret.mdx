**Catégorie :** Web / Injection SQL
**Flag :** `DVCTF{vitruvianman}`
## 1. Introduction
Jacques Saunière, conservateur en chef du Louvre, est décédé dans des circonstances mystérieuses. Il aurait laissé un message pour sa petite-fille Sophie sur l'intranet du musée. L'objectif est d'accéder au portail interne en exploitant les vulnérabilités de l'application pour extraire ce secret.
## 2. Concepts clés
* **SQL injection (SQLi) :** Vulnérabilité permettant d'interagir avec la base de données backend via des paramètres non filtrés dans les requêtes utilisateur.
* **Extraction de données via UNION-based SQLi :** Technique consistant à combiner les résultats d'une requête légitime avec une requête malicieuse pour extraire des informations système ou utilisateur.
* **Hash cracking (SHA-256) :** Utilisation de dictionnaires (wordlists) pour retrouver le mot de passe original à partir d'une empreinte chiffrée.
* **LFI (Local File Inclusion) :** Vulnérabilité permettant de lire des fichiers locaux sur le serveur en manipulant les chemins d'accès fournis en paramètre.
## 3. White-boxing et analyse
L'analyse de la page de recherche (`/search`) révèle une injection SQL via le paramètre `q` (``/search?q=Vinci``). L'injection d'une simple apostrophe (`'`) génère une erreur de syntaxe verbeuse, confirmant l'absence de sanitisation et le moteur utilisé :

```text
ERROR 1064 (42000): You have an error in your SQL syntax; check the manual that corresponds to your MariaDB server version for the right syntax to use near 
''%' OR salle LIKE '%Vinci'%' OR type LIKE '%Vinci'%' OR epoque LIKE '%Vinci'%''
````

Cette erreur est révélatrice de plusieurs choses:
- On comprend que la requête effectue plusieurs `LIKE` imbriqués, rendant l'injection possible par brisure de chaîne.
- Le message confirme l'utilisation d'un serveur **MariaDB**.
- Le comportement différent lors de l'ajout d'une apostrophe prouve que le paramètre `q` est injecté directement dans la requête sans filtrage.

L'automatisation via `sqlmap` permet d'énumérer les bases de données et d'extraire la table `personnel_credentials`. Les hashs SHA-256 des membres du personnel sont récupérés, et le mot de passe de Jacques Saunière est identifié après une attaque par dictionnaire :

```sh
john --format=raw-sha256 --wordlist=/usr/share/wordlists/rockyou.txt hashes.txt
```

- **Mot de passe :** `vitruvianman`
## 4. Résolution
Une fois authentifié avec le compte `j.sauniere` sur `/login`, l'accès au tableau de bord est effectif. Le challenge étant axé sur l'infiltration initiale du portail intranet via l'injection SQL et le cassage de hash, l'accès réussi à l'espace personnel du conservateur, confirmant ainsi le mot de passe (`vitruvianman`), valide la réussite de cette étape.

Le flag final est obtenu en soumettant le mot de passe découvert au format requis : 
* **Flag :** `DVCTF{vitruvianman}`

*(Note : La découverte de traces laissées par Saunière dans la base de données (`logs_acces`) concernant un fichier caché `.message_pour_sophie` constitue l'amorce du challenge suivant.)*
